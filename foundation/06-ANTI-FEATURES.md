# Anti-Features

An anti-feature is something we explicitly reject. These are products or features we will never build, as they dilute Chronicle's core value proposition and introduce clutter.

---

## ❌ 1. Chatbot-Only Product
We are not building a generic ChatGPT wrapper or a floating chat bubble that sits on a screen. 
* While Chronicle has a conversational interface, its core value is the reconstruction of structured timelines, graphs, and evidence citations.
* The product must offer keyboard-driven search and structured visual representations of context, not just text responses.

---

## ❌ 2. Dashboard-First Product
We will not build decorative dashboards, executives charts, or KPI trackers.
* Developers hate decorative, slow dashboards.
* Chronicle is an active utility for engineers. Every screen must contain dense, actionable information, not vanity metrics.

---

## ❌ 3. Ticket Management
We are not building a replacement for Linear, Jira, or GitHub Issues.
* Chronicle is not a task tracking or ticketing system.
* We do not manage sprints, assign tasks, or track project progress. We ingest ticket metadata only to reconstruct memory and decisions.

---

## ❌ 4. Documentation Replacement
We are not asking engineers to write more documents, wikis, or blog posts.
* Chronicle does not seek to replace Confluence or Notion as a place to write manuals.
* Instead, Chronicle reconstructs understanding from the *byproducts* of existing developer work (commits, code, chat threads, pull requests).

---

## ❌ 5. Notion Clone
We will not build a document editor, team workspace, database builder, or general-purpose wiki.
* Chronicle is not an editor.
* We reject Notion’s slow, block-based, mouse-heavy document model.

---

## ❌ 6. Jira Clone
We will not build scrum boards, burndown charts, backlog management tools, or complex workflow configurations.
* Chronicle is a knowledge and memory engine, not a project management platform.

---

## ❌ 7. Confluence Clone
We will not build hierarchical page trees, static wiki articles, meeting notes templates, or comment sections on static pages.
* Our interface is dynamic, evidence-first, and timeline-driven.
